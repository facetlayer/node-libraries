# Error handling

This library uses a normalized shape for errors called `ErrorDetails`. Using a
plain-object representation (rather than `Error` instances) makes errors easy
to serialize, forward between processes, attach extra context to, and chain
together when one error causes another.

## The `ErrorDetails` object

`ErrorDetails` is the canonical representation of an error in this library.
Every `fail` event, every `log_error` event, and every value returned from
`captureError()` is shaped as an `ErrorDetails`.

```typescript
interface ErrorDetails {
    // Unique ID assigned to this error when it's captured.
    // Auto-generated by captureError() if not already set.
    errorId?: string;

    // Short enum-like string used to categorize the error.
    // Examples: 'not_found', 'unhandled_exception', 'generic_error',
    // 'unknown_error', or your own application-specific categories.
    errorType?: string;

    // Human-readable error message.
    errorMessage: string;

    // Stack trace (captured from Error.stack when available).
    stack?: any;

    // Previous error that caused this one. Use this to chain errors
    // while preserving the lower-level cause.
    cause?: ErrorDetails;

    // Arbitrary related information about the error. Each entry is a
    // record of string keys to string values, attached as extra context
    // (such as request IDs, user IDs, file paths, etc.).
    related?: Array<Record<string, string>>;
}
```

`errorMessage` is the only required field. Everything else is optional, but
most errors produced by this library will have at least `errorId`, `errorType`,
and `stack` filled in.

## `captureError(error, related?)`

`captureError()` is the main entry point for converting an arbitrary error
value into an `ErrorDetails` object. It accepts several input shapes and
normalizes them:

| Input | Behavior |
| --- | --- |
| `Error` instance | Copies `message` into `errorMessage`, copies `stack`, assigns a fresh `errorId`, and guesses an `errorType` (e.g. `not_found` if the message starts with `Not found:`, otherwise `unhandled_exception`). |
| `ErrorWithDetails` instance | Unwraps the inner `errorItem`, preserving any existing `errorId`, and merges `related` context. |
| `ErrorDetails` plain object | Passes fields through, filling in `errorId` and `errorType` if missing, and appending any new `related` context. |
| `string` | Wraps the string as `errorMessage`, assigns a fresh `errorId`, and sets `errorType` to `generic_error`. |
| Falsy | Returns a placeholder with `errorMessage: 'Unknown error'` and `errorType: 'unknown_error'`. |

The optional second argument, `related`, lets you attach extra context at
capture time. It will be concatenated onto any existing `related` entries on
the input error.

```typescript
import { captureError } from '@facetlayer/streams';

try {
    doSomething();
} catch (e) {
    const details = captureError(e, [{ requestId: '1234', userId: 'abc' }]);
    console.log(details.errorId, details.errorMessage, details.related);
}
```

## Sending errors through a `Stream`

A `Stream` terminates with either a `done` event (success) or a `fail` event
(failure). The `fail` event always carries an `ErrorDetails` object:

```typescript
interface StreamFail {
    t: c_fail;
    error: ErrorDetails;
}
```

### `stream.fail(error)`

Closes the stream with a `fail` event. The `error` argument can be either an
`Error` or an `ErrorDetails` — `Error` instances are automatically passed
through `captureError()`.

```typescript
const stream = new Stream();

try {
    stream.item(loadRow());
    stream.done();
} catch (e) {
    stream.fail(e); // Error gets normalized to ErrorDetails automatically
}
```

### `stream.closeWithError(error)`

Same as `fail()`, but is a no-op if the stream is already closed. Useful in
cleanup paths where you want to fail the stream only if it hasn't finished
yet.

### `stream.logError(error)`

Emits a `log_error` event without closing the stream. Use this when you want
to surface a non-fatal error alongside the items you're producing. The
`error` argument here must already be an `ErrorDetails`.

## Receiving errors from a `Stream`

### Via `listen()`

The `fail` callback receives a fully-formed `ErrorDetails` object:

```typescript
stream.listen({
    item: (row) => process(row),
    done: () => console.log('all done'),
    fail: (err) => {
        console.error(`[${err.errorType}] ${err.errorMessage} (id=${err.errorId})`);
        if (err.related) {
            for (const entry of err.related) {
                console.error('  related:', entry);
            }
        }
    },
});
```

### Via `pipe()`

With `pipe()` you get the raw event, so match on `c_fail`:

```typescript
import { c_fail } from '@facetlayer/streams';

stream.pipe((evt) => {
    if (evt.t === c_fail) {
        handle(evt.error); // evt.error is ErrorDetails
    }
});
```

### Via promises and iterators

The promise-based and iterator-based consumers (`promiseItems()`,
`promiseItem()`, `wait()`, `forEach()`, `takeItemsSync()`, and
`[Symbol.asyncIterator]()`) all reject / throw using `toException()`, which
wraps the `ErrorDetails` in an `ErrorWithDetails` instance. This means the
original error fields are preserved and can be recovered:

```typescript
import { ErrorWithDetails } from '@facetlayer/streams';

try {
    const items = await stream.promiseItems();
} catch (e) {
    if (e instanceof ErrorWithDetails) {
        const details = e.errorItem; // ErrorDetails
        console.error(details.errorType, details.errorMessage);
    } else {
        throw e;
    }
}
```

Because `ErrorWithDetails` extends `Error`, it can still be thrown, caught,
logged, and passed back to `captureError()` without losing its data.

## `errorDetailsToString(error)`

Formats an `ErrorDetails` into a human-readable single-line-plus-stack string.
Useful for log output:

```
error (not_found): user with id 42 was not found
Stack trace: ...
```

## `ErrorWithDetails`

`ErrorWithDetails` is a subclass of `Error` that carries a full `ErrorDetails`
object on its `errorItem` property. The library produces these when it needs
to raise an exception from a stream consumer (for example, inside `await
stream.promiseItems()`).

You can construct one explicitly if you want to throw an error that carries
full details:

```typescript
import { toException } from '@facetlayer/streams';

throw toException({
    errorType: 'not_found',
    errorMessage: 'user not found',
    related: [{ userId: 'abc' }],
});
```

If you later pass this exception to `captureError()`, the details are
recovered unchanged (including `errorId`, `errorType`, `related`, etc.).

## Global unhandled error listener

Some code paths inside the library catch errors that can't be attributed to
any specific stream (for example, an error from an async callback after the
downstream consumer has already stopped listening). These errors are routed
to a global error listener so they can still be observed.

```typescript
import { startGlobalErrorListener, recordUnhandledError } from '@facetlayer/streams';

const errors = startGlobalErrorListener();
errors.onItem((err) => {
    console.error('unhandled stream error:', err.errorMessage);
});

// Your own code can also push into the global listener:
recordUnhandledError(new Error('something went wrong in a detached task'));
```

`startGlobalErrorListener()` returns a `Stream` of `ErrorDetails` items that
you can consume just like any other stream.

## Chaining errors with `cause`

When one error triggers another, set `cause` on the new error to preserve the
lower-level context:

```typescript
try {
    await readConfig();
} catch (e) {
    stream.fail({
        errorType: 'startup_failed',
        errorMessage: 'server failed to start',
        cause: captureError(e),
    });
}
```

Consumers can walk the `cause` chain to surface the full history of what went
wrong.

## Attaching related context

Use `related` to attach structured context that isn't part of the error
message itself — things like request IDs, user IDs, file paths, or retry
counts. `captureError()` will merge any new `related` entries you pass as its
second argument with whatever is already on the error:

```typescript
const details = captureError(e, [
    { requestId: req.id },
    { route: req.path },
]);
```

`related` is an array of records (not a single record) so that multiple
layers of the stack can each append their own context without overwriting
what came before.
